You are implementing the fix for **bug6** — *bug6 — OData filter parser rejects single-character identifiers. In cf-modkit-odata, parsing a filter whose field name is a single character (e.g. `x eq 1`) fails with a parse error: 'error at 1:2: expected [a..z|A..Z|_|0..9]'. Multi-character identifiers parse fine. The identifier grammar rule requires a first char followed by ONE OR MORE additional chars, so identifiers of length 1 are wrongly rejected. Expected: a one-character identifier is valid — `parse_str("x eq 1")` must succeed and yield an equality comparison of identifier `x` with the number 1. Source: libs/modkit-odata/src/odata_parse.rs (the PEG `identifier` rule). Tests: cargo test -p cf-modkit-odata --lib.*
inside the worktree at `/Users/brad/code/gears-rust/.worktrees/marathon-bug6-gpt`.

The proposing room produced this proposal:

> In the PEG grammar's `identifier` rule in `libs/modkit-odata/src/odata_parse.rs`, change the trailing identifier-character repetition from one-or-more (`+`) to zero-or-more (`*`) while keeping the existing first-character rule unchanged. Keep or add the regression test `single_character_identifier` in `libs/modkit-odata/src/odata_parse_tests.rs` asserting that `parse_str("x eq 1")` returns `Expr::Compare(Expr::Identifier("x"), Equal, Expr::Value(Value::Number(1)))`.

**Root cause:** The `identifier` rule at `libs/modkit-odata/src/odata_parse.rs:98-99` requires a first identifier character followed by `['a'..='z'|'A'..='Z'|'_'|'0'..='9']+`. Because `+` means one or more, the grammar accidentally makes the minimum identifier length two characters. When parsing `x eq 1`, the parser consumes the initial `x` as the first character and then fails at column 2 because the next character is a space instead of another identifier character.

**Affected files:** libs/modkit-odata/src/odata_parse.rs, libs/modkit-odata/src/odata_parse_tests.rs

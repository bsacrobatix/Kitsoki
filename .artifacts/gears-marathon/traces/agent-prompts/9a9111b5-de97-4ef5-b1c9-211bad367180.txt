You are reviewing the implementation of the fix for **bug5** —
*bug5 — resource-group wrongly forces the RG type-registry prefix onto allowed-membership type codes. In cyberware-resource-group, TypeService::create_type / update_type validate each allowed_membership_types entry with validation::validate_type_code, which enforces the RG namespace prefix (gts.cf.core.rg.type.v1~). But membership resource types are EXTERNAL domain types (e.g. gts.cf.core.idp.user.v1~) per DESIGN.md and ADR-001 Type Map; they must NOT be forced into the RG type-registry namespace. Effect: callers must wrap external types in the RG prefix purely to pass validation — a contract divergence. Expected: allowed-membership type codes accept external (non-RG-prefixed) domain types, while still rejecting empty and over-length (>1024) codes; RG-prefixed codes remain accepted for backwards-compat. add_membership/remove_membership are already correct and must stay unchanged. Source: modules/system/resource-group/resource-group/src/domain/ (validation.rs, type_service.rs). Tests: cargo test -p cyberware-resource-group --test domain_unit_test.*.

The proposed fix was:

> ## Bug
`TypeService::create_type` and `TypeService::update_type` reject registered external domain type codes in `allowed_membership_types`, such as `gts.cf.core.idp.user.v1~...`, because they apply the resource-group type registry prefix validator.

## Cause
`validation::validate_type_code` enforces `gts.cf.core.rg.type.v1~` and is correctly used for RG type codes and parent types. The membership loops in `type_service.rs` reuse that RG-specific helper, so external membership types are rejected before `type_repo.resolve_ids` can verify that the type row exists.

## Fix
Add a membership-specific validation helper in `validation.rs` that preserves normalization plus empty and 1024-character length rejection but does not require the RG prefix. Use it only for `allowed_membership_types` in `create_type` and `update_type`. Keep parent/type validation and `add_membership`/`remove_membership` unchanged.

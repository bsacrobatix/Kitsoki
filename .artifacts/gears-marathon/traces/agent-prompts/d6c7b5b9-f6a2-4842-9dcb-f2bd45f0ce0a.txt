## Summary
bug5 was a contract divergence in `cyberware-resource-group`: `TypeService::create_type` and `TypeService::update_type` validated `allowed_membership_types` with the resource-group type-registry validator, which forced the `gts.cf.core.rg.type.v1~` prefix onto membership resource type codes. Per the design evidence, allowed membership types are external domain resource types such as `gts.cf.core.idp.user.v1~`, so they must not be constrained to the RG type namespace.

## Fix
The fix introduced and applied membership-specific type-code validation for `allowed_membership_types`. That validation still rejects empty entries and codes longer than 1024 characters, but no longer enforces the RG namespace prefix. Existing RG-prefixed codes remain accepted for backwards compatibility. RG type codes and parent type codes still use the stricter RG `validate_type_code`, and `add_membership` / `remove_membership` behavior was left unchanged.

The relevant tests passed, including create/update acceptance for external membership types and rejection coverage for empty and over-length entries. Final `git status --short` was clean.

## Future Approach

## Artifact title

> bug4 — CanonicalError common-library conversions verified RED

## Artifact body

## What is broken
`cf_modkit_errors::CanonicalError` cannot currently be produced from common library errors. Code that calls `CanonicalError::from(std::io::Error)`, `CanonicalError::from(serde_json::Error)`, or uses `?` to propagate those errors into `Result<_, CanonicalError>` does not compile.

## Reproduction
I added behavior-level tests to `libs/modkit-canonical-errors/tests/error.rs` covering both direct conversion and `?` propagation for `std::io::Error` and `serde_json::Error`. The tests assert the expected public behavior: IO errors become status 500/title `Internal`, serde JSON errors become status 400/title `Invalid Argument`, and the serde detail matches the serde error message.

Run:

```sh
cargo test -p cf-modkit-canonical-errors --test error
```

## Evidence
The command is RED on the current tree. Rust reports that `CanonicalError::from(err)` expects an existing `CanonicalError` rather than an IO/serde error, and that `?` cannot convert `std::io::Error` or `serde_json::Error` because the required `From` impls are missing.

## Implicated components
`cf-modkit-canonical-errors` is the library crate under test. The affected API surface is `libs/modkit-canonical-errors/src/error.rs`, where `CanonicalError` is defined and where ergonomic `From` conversions for common library errors are absent.

## Decision criteria

- **accept** — `bug_verified` is true, evidence is concrete (test, script,
  recorded sequence), `involved_components` are plausible, and the
  reproduction can be re-run by another engineer from the steps alone.
- **refine** — the artifact is on the right track but missing something
  important (e.g. no executable repro, fuzzy expected/actual outcome,
  involved_components feel made-up). Put the specific gap in `reason`;
  the next iteration's prompt will see it as feedback.
- **restart_from** — the wrong bug is being reproduced; the engineer
  misread the ticket. Reset to idle / reproducing.
- **quit** — this bug is unreproducible or out-of-scope (e.g. user error,
  duplicate of a closed ticket). Rare.
- **uncertain** — you cannot judge from the artifact alone (e.g. it
  refers to a service you have no context on). Yield to a human.

Set `confidence` honestly. Confidence below the project threshold (default
0.8) means a human will be asked even in `llm_then_human` mode; in strict
`llm` mode an uncertain verdict leaves the state held until an operator
intervenes.

## Output

Submit a `judge_verdict` (see `schemas/judge_verdict.json`):
`{ verdict, intent, reason, confidence }`. Keep `reason` actionable —
the engineer reads it on `refine`.

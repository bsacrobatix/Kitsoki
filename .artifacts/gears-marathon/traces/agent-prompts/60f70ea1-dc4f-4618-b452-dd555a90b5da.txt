## Summary

`cf_modkit_errors::CanonicalError` could not be constructed ergonomically from common library errors. Code returning `Result<_, CanonicalError>` could not use `?` on `std::io::Error` or `serde_json::Error`, and callers lacked direct `CanonicalError::from(...)` conversions.

The fix added `From<std::io::Error>` and `From<serde_json::Error>` implementations in `libs/modkit-canonical-errors/src/error.rs`. IO errors now map to the existing internal canonical error path with status 500 and title `Internal`. serde_json errors now map to invalid argument with status 400, title `Invalid Argument`, and the serde error message as detail.

Regression coverage was added in `libs/modkit-canonical-errors/tests/error.rs` for direct conversion and `?` propagation for both source error types. Targeted validation passed: `cargo build -p cf-modkit-canonical-errors`, `cargo test -p cf-modkit-canonical-errors --test error` with 11 passed, and `cargo test -p cf-modkit-canonical-errors` across the package test suite. A broader workspace test was attempted but stopped in unrelated `libz-ng-sys` build-script setup because cmake was unavailable.

No refinement cycles were required. For similar conversion bugs, the useful pattern is to validate both the visible canonical response shape and compile-time ergonomics through a small `?` propagation test, not just direct constructor assertions.

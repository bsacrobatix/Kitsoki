You are reviewing the implementation of the fix for **bug6** —
*bug6 — OData filter parser rejects single-character identifiers. In cf-modkit-odata, parsing a filter whose field name is a single character (e.g. `x eq 1`) fails with a parse error: 'error at 1:2: expected [a..z|A..Z|_|0..9]'. Multi-character identifiers parse fine. The identifier grammar rule requires a first char followed by ONE OR MORE additional chars, so identifiers of length 1 are wrongly rejected. Expected: a one-character identifier is valid — `parse_str("x eq 1")` must succeed and yield an equality comparison of identifier `x` with the number 1. Source: libs/modkit-odata/src/odata_parse.rs (the PEG `identifier` rule). Tests: cargo test -p cf-modkit-odata --lib.*.

The proposed fix was:

> ## Bug
`cf-modkit-odata` rejects valid filters whose field identifier has one character, such as `x eq 1`.

## Cause
The PEG `identifier` rule in `libs/modkit-odata/src/odata_parse.rs:98-99` requires a valid first character plus one or more additional identifier characters. That makes the effective minimum identifier length two, so `x eq 1` fails at the space after `x`.

## Fix
Change only the suffix repetition in the `identifier` rule from `+` to `*`, keeping the first-character restriction unchanged. Retain the regression test in `libs/modkit-odata/src/odata_parse_tests.rs` for `parse_str("x eq 1")` producing an equality comparison against numeric value `1`.

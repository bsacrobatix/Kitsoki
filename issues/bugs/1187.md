## Bug Verified: `studioImportResolver` hard-errors on library stories when `--stories-dir` is set

### What is broken

`studioImportResolver` in `cmd/kitsoki/mcp.go` builds a closure over the base resolver. When `--stories-dir` is set and the import resolver is called with `override=true` for a story **not present in storiesDir**, the closure immediately returns a hard error:

```
--stories-dir=<dir>: story "dev-story" not found (looked for <dir>/dev-story/app.yaml): stat …: no such file or directory
```

The `ImportResolver` contract (documented in `internal/app/imports.go`) states that `override=true` should return `("", nil)` to fall through to the next tier (on-disk checkout, then embedded library) when no explicit override is configured for that name. The `--stories-dir` wrapper violates this contract: it treats _any_ story absent from `storiesDir` as a hard error, regardless of whether `$KITSOKI_REPO`, the embedded library, or an on-disk checkout could satisfy it.

### Affected call paths

Both `session.new` and `story.graph` ultimately call `app.LoadWithResolver(storyPath, nil, srv.importResolver)` where `srv.importResolver` is `studioImportResolver(storiesDir)`. Any downstream root story (e.g. a project instance under `.kitsoki/stories/`) that imports `@kitsoki/dev-story` fails to load when `kitsoki mcp --stories-dir <dir>` is running and `dev-story` is not in `<dir>`.

### How reproduced

A minimal deterministic test was written (`cmd/kitsoki/resolver_bug1187_repro_test.go`):
1. Sets `KITSOKI_REPO` to the kitsoki worktree (so a correct fall-through resolves `dev-story`).
2. Creates a `storiesDir` containing only a custom user story (no `dev-story`).
3. Creates a downstream `app.yaml` (in a temp dir outside the checkout) that declares `imports.core.source: "@kitsoki/dev-story"`.
4. Calls `app.LoadWithResolver(downstreamApp, nil, studioImportResolver(storiesDir))`.
5. Asserts the load **succeeds** — which it does NOT on the unfixed tree.

### Root cause (buggy line)

```go
// cmd/kitsoki/mcp.go, studioImportResolver — the override=true branch:
if _, err := os.Stat(candidate); err != nil {
    // BUG: hard error instead of falling through
    return "", fmt.Errorf("--stories-dir=%s: story %q not found (looked for %s): %w",
        storiesDir, name, candidate, err)
}
```

### Fix (verified by temporarily applying it)

Replace the hard error with a delegation to the base resolver:
```go
return base(name, importerDir, override)
```

This lets `$KITSOKI_REPO`, kit-dev overrides, and the embedded library satisfy imports that `storiesDir` doesn't cover — which is the correct layering.

### Evidence

- Test file committed: `cmd/kitsoki/resolver_bug1187_repro_test.go` at commit `52e455b2`
- RED on unfixed tree: error is `imports.core: source "@kitsoki/dev-story": --stories-dir=…: story "dev-story" not found`
- GREEN after one-line fix: load succeeds and `def.States["core"]` is populated
